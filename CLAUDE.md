# Multiplayer Markdown

Self-hosted, real-time collaborative sync for Obsidian vaults, with per-folder permissions — built from scratch because no existing tool (Relay, Peerdraft, YAOS, Team Relay, etc.) combines: real Obsidian/markdown files + true CRDT live co-editing + free/self-hosted/unlimited + reliable per-folder privacy + reasonable maturity. Full research trail is in the vault root: `../Self-Hosted Obsidian Multiplayer Research.md`.

**User's locked-in requirements/decisions (do not re-litigate):**
- Real Obsidian vault, plain markdown files on disk — not a proprietary format.
- True real-time simultaneous multi-user editing (live cursors, CRDT merge), not background sync.
- Self-hosted on a **VPS/Docker** (not serverless/Cloudflare Workers).
- Auth: **simple admin-managed accounts** (admin adds users via the CLI or the `/api/admin` web UI, no self-service signup).
- Permissions: **per top-level shared folder** only (not arbitrary nested ACLs). Each folder = one "room" with its own viewer/editor member list.
- Free/self-hosted, no subscription, no artificial user caps.

Full architecture/build-phase plan: `C:\Users\lukes\.claude\plans\sunny-sprouting-volcano.md` (should still exist, but this file is self-contained enough to not strictly require it).

## Architecture

One Yjs `Y.Doc` per shared top-level folder ("room") — not one per vault (too coarse for ACLs) and not one per file (too much coordination overhead). A room's doc:

```
Y.Doc "room:<roomId>"
 ├── Y.Map "files"           // relativePath -> Y.Text (file content), relative to the room's folder root
 ├── Y.Map "attachmentsMeta" // relativePath -> { hash, size, mtime }  (Phase 5, built)
 └── Y.Map "trash"           // soft-deleted path -> { deletedAt, snapshot }  (Phase 7, not built yet)
```

**Server**: `@hocuspocus/server` (Node/TS) on `PORT` (default 4444) for the WebSocket/CRDT protocol, plus a **separate plain `node:http` server on `HTTP_API_PORT`** (default 4445, no Express) for `POST /api/login`, `GET /api/rooms`, attachment routes, and (Phase 7) a full `/api/admin/*` REST surface plus a self-contained `GET /api/admin` web UI for managing users/rooms without the CLI. `onAuthenticate` is the single place room-permission logic lives — JWT verification (with `token_version` for instant revocation), room-membership lookup, `connectionConfig.readOnly = true` for viewers (confirmed server-side: Hocuspocus drops that connection's Yjs update messages entirely, never applying them). Admin routes are gated separately by `authenticateAdmin` (same JWT/`token_version` re-check, but checking `is_admin` instead of room membership — JWTs carry no admin claim, so this is a fresh DB lookup on every request, not a cached/client-trusted flag). `onLoadDocument`/`onStoreDocument` hydrate/persist each room to real `.md` files on disk under `/vaults/<roomId>/`. SQLite (`better-sqlite3`) holds `users`/`rooms`/`room_members`; passwords hashed with `argon2id`. An optional `ADMIN_USERNAME`/`ADMIN_PASSWORD` env-var pair bootstraps the first admin account on startup (idempotent — skipped once that username exists, never resets a password), so a fresh deployment needs zero terminal access to become usable.

**Plugin**: standard Obsidian plugin (esbuild). Live CM6 co-editing via `y-codemirror.next` + `@hocuspocus/provider` for whichever file is currently open; a `FileSyncEngine` watches vault create/modify/delete/rename events for closed files and reconciles them via diff (not blind overwrite). A `RoomManager` owns one provider + one `FileSyncEngine` per linked room (a user can be a member of several rooms/folders at once); `CollabBinder` looks up which active room (if any) owns a given file path via `RoomManager.findRoomForPath` rather than a single hardcoded folder.

**Monorepo** (pnpm workspaces): `packages/sync-core` (shared Yjs helpers, used by both server and plugin — the one place file↔Y.Text logic lives), `packages/server`, `packages/plugin`.

**What works right now:** real accounts with real per-room permissions. An admin creates users and rooms through the `/api/admin` web UI (or the CLI, which still works identically — both operate on the same SQLite rows), grants viewer/editor roles per room, and users log into the plugin (username/password → JWT, stored in the plugin's own `data.json`) and link any room they're a member of to a local folder via the settings tab. That "local folder" field must be a vault-relative path (e.g. `Shared`), not an OS filesystem path — see gotcha #7 below, found from a real "nothing is syncing" report. Multiple rooms can be linked simultaneously — each gets its own live provider connection and `FileSyncEngine`. Editors get full real-time co-editing; viewers get read-only client-side (CM6 `EditorState.readOnly`) **and** the server actually enforces it independently (`onAuthenticate` sets `connectionConfig.readOnly`, which makes Hocuspocus silently drop that connection's write messages without ever touching the document). Non-members are refused the connection outright. Revoking a user (`user revoke`) invalidates their JWT immediately via `token_version`, without waiting for expiry.

**Verified two ways:**
1. Headless (`verify:auth`, 8 checks): editor writes succeed; viewer writes never reach the editor's copy of the document (proving server-side rejection, not client-side hiding); a non-member's connection never reaches `synced`; a revoked token is rejected immediately.
2. Real Obsidian, two accounts (alice=editor, bob=viewer) on the same room: alice creates a file → bob receives it; bob edits his local copy directly via the vault API (bypassing the UI entirely) → alice's copy and the server's persisted disk copy are both untouched — the tampering never left bob's machine; bob's CM6 editor state is `readOnly: true`, alice's is `false` and she can still type live.

Re-implementing proper rotating refresh tokens is deferred to Phase 7 polish; it's a UX nicety (avoiding re-logins), not a security property — the actual security boundary (`onAuthenticate` + `token_version`) doesn't depend on it.

## Phase 5 — binary attachments (done)

Yjs isn't suited to large blobs, so attachments (images, PDFs, etc.) live entirely outside the CRDT layer, exactly as planned:

- **Server**: `attachments.ts` (`sha256Hex`, `attachmentExists`/`readAttachment`/`writeAttachment` — content-addressed, one store per room at `data/attachments/<roomId>/<hash>`; `writeAttachment` re-hashes the uploaded bytes and refuses to store them if they don't match the claimed hash, so a client can't poison the content-addressed store). Three new routes on the existing HTTP API (`http-api.ts`): `GET`/`HEAD`/`PUT /api/rooms/:roomId/attachments/:hash`, all gated by `authenticateForRoom` (the same membership-check helper `onAuthenticate` uses, extracted out during this phase so both paths share one implementation) — `PUT` additionally requires the `editor` role, matching the same read-only rule enforced over WebSocket.
- **sync-core**: `attachmentsMeta` Y.Map helpers (`getAttachmentMeta`/`setAttachmentMeta`/`deleteAttachmentMeta`/`listAttachmentPaths`) — stores only `{hash, size, mtime}` per relative path, not the bytes.
- **Plugin**: `attachment-client.ts` (`sha256Hex` via Node's `crypto` — available in Obsidian's renderer since Obsidian has nodeIntegration; `uploadAttachment`/`downloadAttachment`/`attachmentExistsRemotely` via `requestUrl`). `FileSyncEngine` now branches on file extension: `.md` files use the existing text-sync path, everything else routes through `syncLocalAttachmentToRemote`/`materializeAttachment` — hash-compare before uploading/downloading (skip if content unchanged or already present), separate shallow `observe()` on the attachments map (not `observeDeep` — attachment metadata is one atomic value per `.set()` call, unlike the two-phase markdown file case that needed `observeDeep`).

**Verified end-to-end in real Obsidian** (alice=editor, bob=viewer, same setup as Phase 4): alice creates a 300-byte binary file → bob receives it byte-for-byte identical; confirmed the actual blob lands on the server's disk at the expected content-addressed path; bob (viewer) attempts to overwrite the file locally → upload correctly rejected server-side (403), alice's copy and the room's official `attachmentsMeta` hash are both completely untouched by bob's tampering; alice deletes the file → it disappears on bob's side too.

**Design decision: two ports, not one shared HTTP server.** The plan originally called for Express sharing Hocuspocus's HTTP server. Investigated `onRequest` (Hocuspocus's hook for the same underlying `http.Server`) as a way to avoid a second port, but its default request handler unconditionally writes a "Welcome to Hocuspocus" response after hooks run unless the hook throws — and throwing propagates as an unhandled rejection risk. Not worth the fragility for 2 routes. Went with a second plain `node:http` server (`HTTP_API_PORT`, default 4445) instead — no Express dependency needed. Phase 6's Caddy reverse proxy can still expose both under one public domain via path-based routing.

**Note found while building Phase 3 (unrelated pre-existing issue, now fixed):** `packages/server/src/verify-sync.ts` had a type error that only surfaced once `tsc` was run across the whole server package for the first time — `HocuspocusProviderConfiguration` doesn't actually allow `WebSocketPolyfill` alongside a plain `url`, only alongside `websocketProvider`. Tried switching to the "correct" `websocketProvider` shape and it broke the actual connection (client never synced) — so the fix was to keep the original working runtime shape (`url` + `WebSocketPolyfill`) and cast past the overly-strict published type, rather than "fixing" it into a shape that doesn't actually work. If you see this error again, don't restructure the provider construction — just cast.

## Phase 6b — Cloudflare Tunnel / Unraid: single all-in-one image (✅ done, verified against a real live deployment)

Option A (Cloudflare Tunnel) went through three real designs before landing on the current one — each earlier one was tried against an actual user deployment and rejected for a concrete reason, not just theorized away:

1. **Two Cloudflare Public Hostname rules, two ports** (`/api/*` → HTTP API port, catch-all → WebSocket port), both pointed at the same tunnel. This reliably trips a real Cloudflare **dashboard** bug: adding the second rule under one tunnel hostname throws "A, AAAA, or CNAME record with that host already exists," because the dashboard tries to (re)create a DNS record the first rule already created. Workaround exists (delete the DNS record, then re-save) but it's a real, repeatable rough edge for anyone following this path.
2. **A 3-container Docker Compose stack** (`sync-server` + a plain-HTTP `caddy-tunnel` + `cloudflared` itself, all sharing one Compose network so they resolve each other by container name) — worked, avoided the dashboard bug entirely, but the user explicitly rejected running multiple containers: "I don't want to have multiple containers, I want a single container."
3. **Final design — one image, two Dockerfile targets:** `packages/server/Dockerfile` now has a second final stage, `allinone` (alongside the original `server` stage, which Option B still uses unchanged). It bundles the `caddy` binary (`COPY --from=caddy:2 /usr/bin/caddy /usr/bin/caddy`) and a baked-in `packages/server/docker/Caddyfile` into the same image as the Node server, run together by `packages/server/docker/entrypoint.sh` — a ~15-line bash script that backgrounds both processes, traps `SIGTERM`/`SIGINT` to forward to both, and uses `wait -n` so if *either* process dies the whole container exits (letting Docker's restart policy relaunch both together, never leaving one alive without the other). Since Caddy and Node share one network namespace in this design, the baked-in Caddyfile talks to plain `localhost:4444`/`localhost:4445` — no env var indirection, no bind-mounted config, nothing for the user to hand-create. Cloudflare's tunnel (run entirely separately by the user — this project never runs `cloudflared` itself) only ever needs **one** Published Application route pointed at **one** port.

**CI** (`.github/workflows/docker-publish.yml`) builds and pushes *two* images from that one Dockerfile in the same job, via `target: server` and `target: allinone` respectively — `ghcr.io/.../multiplayer-markdown-sync-server` (Option B, unchanged) and `ghcr.io/.../multiplayer-markdown` (Option A, the new all-in-one). Docker defaults to the *last* stage in a Dockerfile when no `--target` is given, so appending `allinone` after `server` required adding an explicit `target: server` to the pre-existing build step — without it, that step would have silently started building the wrong image.

`docker-compose.yml`'s earlier `caddy-tunnel`/`cloudflared` services (design #2 above) and the root-level `Caddyfile.tunnel` were deleted entirely once design #3 landed — Option A no longer touches `docker-compose.yml` at all, it's a plain `docker run` or a single Unraid "Add Container" entry. `docs/unraid-compose.md` no longer covers Cloudflare Tunnel for the same reason; it's Option-B-only now (bundled Caddy + a real Let's Encrypt cert, via Compose Manager).

**Real routing gotcha found while designing the admin UI (Phase 7) on top of this:** both Caddy configs — the root `Caddyfile` (Option B) and `packages/server/docker/Caddyfile` (Option A) — only forward the `/api/*` path prefix to the HTTP API port; everything else (the catch-all) goes to the Hocuspocus WebSocket port instead. Any new HTTP route, page or API, **must** live under `/api/*` or it silently gets routed to the wrong port in both deployment modes. This is why `/api/admin` (the web UI page itself, not just its JSON endpoints) lives under `/api/`, even though it's serving an HTML page rather than JSON.

**Real unrelated bug hit during actual deployment (worth remembering if it comes up again):** a domain that had previously been parked/forwarded (via the registrar, Porkbun in this case) can leave a stale redirect — such as a Cloudflare **Bulk Redirect** or legacy **Page Rule** — that intercepts traffic before it ever reaches the tunnel, even after nameservers are correctly pointed at Cloudflare and DNS records look right. It presented as a `302` to a `<domain-with-dashes>.l.ink` URL with `Server: cloudflare` but an `openresty`-branded HTML body — the giveaway that it wasn't this project's Caddy or Cloudflare Tunnel at all, but something upstream in the account's own Rules configuration. **Rules → Redirect Rules** alone isn't enough to check — **Bulk Redirects** and **Page Rules** are separate features in Cloudflare's dashboard and are easy to miss.

**Verified live** (not just headlessly): real Cloudflare Tunnel connector (the user's own, run as a separate Unraid app), real Unraid container from the published `ghcr.io/.../multiplayer-markdown` image, real domain. Confirmed via `curl` that `/api/login` returns the correct `401` JSON through the full chain (Cloudflare → tunnel → container's internal Caddy → Node HTTP API), visible in the `Via: 1.1 Caddy` response header.

**Real data-loss bug found from an actual user report (fixed):** `DB_PATH`/`VAULTS_ROOT`/`ATTACHMENTS_ROOT` (`index.ts`) each default to a *relative* `./data/...` path when unset, resolved against `WORKDIR` (`/prod/server` in both the `server` and `allinone` Dockerfile stages) — not against `/data`. The README's Option A instructions (both the Unraid "Add Container" env-var list and the plain `docker run` example) mount volumes at `/data/db`, `/data/vaults`, `/data/attachments` but never told the deployer to also set those three env vars, so the app was actually reading/writing `/prod/server/data/...` — inside the container's own writable layer, never on the mounted volume. A real deployment lost every room and user the next time the container got recreated (e.g. an image update), *except* the admin account, which silently reappeared because `ADMIN_USERNAME`/`ADMIN_PASSWORD` recreate it idempotently on every boot — that partial survival is exactly what made the bug look like "everything is gone except the admin login" instead of an obvious total-data-loss crash. Fix: `packages/server/Dockerfile`'s `base` stage now sets `ENV DB_PATH=/data/db/collab.sqlite3`, `ENV VAULTS_ROOT=/data/vaults`, `ENV ATTACHMENTS_ROOT=/data/attachments`, so both runtime stages are correct by default even if a deployer's env var list is incomplete. `docker-compose.yml` (Option B) already passed the identical values explicitly, so it was never affected — this was Option A (the all-in-one image) only.

## Phase 7 — Admin web UI (✅ done)

Replaces the CLI-only admin workflow with a single self-contained page, so non-technical self-hosters never need a terminal to create accounts or rooms.

- **`packages/server/src/admin-ui.ts`** (new) — exports one `ADMIN_HTML` template-string constant: login form, users table + add-user form, rooms table + create-room form, and a per-room members panel (grant/revoke, viewer/editor). No new dependency, no bundler, no build step — the whole page (inline `<style>`/`<script>`) is just TS source compiled like everything else. Deliberately renders all user-supplied strings (usernames, labels) via `textContent`/DOM APIs rather than raw HTML interpolation.
- **`packages/server/src/auth.ts`** — new `authenticateAdmin`, same verification chain as `authenticateForRoom` (verify JWT → re-check `token_version` against the DB for instant revocation) but checking `user.is_admin === 1` instead of room membership. JWTs never carry an admin claim, so this is a fresh DB lookup every call — revoking someone's admin status takes effect on their very next request, not just their next login.
- **`packages/server/src/http-api.ts`** — 9 new routes under `/api/admin/*` (user add/list/revoke/delete, room create/list/members/grant/revoke — exact parity with `cli.ts`, nothing more) plus the unauthenticated `GET /api/admin` page route, all gated through one `requireAdmin` helper (missing token → `401`; invalid/expired/revoked/not-admin → `403`, matching the precedent already set by the attachment routes' `authenticateForRoom` handling rather than inventing a finer-grained scheme).
- **`packages/server/src/index.ts`** — optional `ADMIN_USERNAME`/`ADMIN_PASSWORD` env-var bootstrap, idempotent (create-if-username-doesn't-exist, otherwise a no-op) so it's safe to leave set permanently across restarts without ever resetting the password.
- **`packages/server/src/verify-admin-api.ts`** (new, `pnpm run verify:admin`) — 47 headless checks. The security-critical ones: every one of the 9 admin routes rejects a missing token (`401`), a valid *non-admin* token (`403`), a malformed token (`403`), and a revoked admin's stale-but-unexpired token (`403`) — the concrete proof that "hide the button" isn't the security model here.
- **Verified live**: created a real user + room + membership grant through the actual API (not just headless), confirmed the CLI (`user list`/`room list`/`room members`) sees the identical state (same SQLite rows, not a parallel system), then logged in as that real non-admin account and confirmed a genuine `403` from the browser-facing flow, not just the test suite.

## BRAT-compatible plugin releases (✅ done)

The plugin isn't in Obsidian's official Community Plugins store, but it installs and auto-updates via **BRAT** (Obsidian42 - BRAT), which works by pulling `manifest.json` and `main.js` off a GitHub Release's attached assets — not from files sitting in the repo.

- **`.github/workflows/plugin-release.yml`** (new) — triggers on a bare-semver tag push (e.g. `0.1.0`, no `v` prefix — matches Obsidian's own plugin template convention, and deliberately never collides with `docker-publish.yml`'s `v*`-prefixed server-image tags). Builds `sync-core` then the plugin, **fails the build if the pushed tag doesn't match `manifest.json`'s `version` field** (catches releasing without bumping it first — a mismatch here would silently confuse BRAT's update detection), then attaches the built plugin assets to a GitHub Release. As of the "CI + release-asset guard" section below, the shipped-file list (`main.js`, `manifest.json`, `styles.css`) is defined exactly once and used for both an existence check and the actual `gh release create` upload, rather than a separate static list that can silently omit a file (which is exactly what happened before that fix — see the Phase 7 continued section).
- **Gotcha found**: `pnpm/action-setup@v4` fails with "No pnpm version is specified" if the repo has no `packageManager` field and the action isn't given an explicit `version` input. Fixed by adding `"packageManager": "pnpm@11.10.0"` to the repo-root `package.json`.
- First real release cut and verified: tag `0.0.1`, confirmed via `gh release view 0.0.1` that both `main.js` and `manifest.json` are attached. Any future release is just: bump `packages/plugin/manifest.json`'s `version`, tag with that exact same bare version string, push the tag.
- `manifest.json`'s `author`/`authorUrl` were fixed from placeholder values (`"you"`) to real ones, since a BRAT-installed plugin makes these genuinely user-visible for the first time.

## Phase 7 continued — production bug fixes: offline-edit data loss + invisible presence/cursors (✅ fixed, headless + live-DOM verified; not yet released)

A real production deployment (`docs.lukeslibrary.com`, cross-state) surfaced two serious bugs neither local dev testing nor the existing `verify:*` suite had caught:

1. **Offline edits were silently discarded on reconnect, not merged.** Root cause: `sync-core`'s `getOrCreateFileText`/`setFileContent` create a brand-new `Y.Text` for a path if the *local* `Y.Doc` doesn't already have that key. A freshly-created `Y.Doc` (e.g. after an app restart with no local persistence) has no way to know a path's `Y.Text` already exists on the server until it actually syncs — so a client that types offline right after a restart (before ever syncing) seeds its edit into a **second, independent `Y.Text` object assigned to the same `Y.Map` key**. Two independently-created values assigned to the same `Y.Map` key do not merge when the docs sync: Yjs's conflict resolution keeps exactly one and silently discards the other's entire content. Confirmed with a minimal standalone repro before touching any code (two `Y.Doc`s, one seeded via `getOrCreateFileText` before ever exchanging updates, merged via `Y.applyUpdate` both ways — the second doc's entire edit vanished, not just conflicted).
   - **Fix**: `RoomManager.syncToLinkedRooms` (`packages/plugin/src/room-manager.ts`) now creates a `y-indexeddb` `IndexeddbPersistence` bound to each room's `Y.Doc` and awaits `.whenSynced` *before* anything else (including awareness setup and `FileSyncEngine.initialScan()`) touches the doc. This means a restarted client always has the real, previously-synced `Y.Text` for every known file loaded from local IndexedDB before it ever calls `setFileContent` — so offline edits reconcile into the existing object (a diff-merge) instead of colliding with it. `ActiveRoom` now carries `indexeddbPersistence` for cleanup in `destroyAll()`/room removal.
   - **Verified**: `packages/server/src/verify-offline-merge.ts` (`pnpm run verify:offline-merge`) — real `@hocuspocus/server`, real `y-indexeddb` (backed by `fake-indexeddb`, the standard IndexedDB shim for Node), real `sync-core` helpers. Simulates: client syncs once, fully destroys itself (full app-restart), creates a brand-new `Y.Doc` hydrated from the *same* IndexedDB persistence name, types an edit while still offline, then reconnects. A negative-control variant (same test, but the restarted client points at a *different* persistence name, simulating the pre-fix behavior of no local persistence at all) was run once to confirm the bug reproduces without the fix — the offline edit was completely lost, proving the fix is what prevents it, not a coincidence.
2. **Presence dots and remote CM6 cursors never rendered — only the "shared room" link icon showed.** Two independent, compounding causes:
   - `.github/workflows/plugin-release.yml` only ever attached `packages/plugin/main.js` and `manifest.json` to the GitHub Release — **`styles.css` was never shipped to a BRAT install**, so `.mm-presence-dot`/`.mm-shared-icon` (defined in `styles.css`) had zero layout rules in production. This was never caught locally because `sync-test-vaults.mjs` always copies `styles.css` unconditionally if it exists in the build output.
   - Separately, `remote-selections.ts`'s `YRemoteCaretWidget` (the CM6 remote-cursor widget) has *always* referenced `cm-ySelectionCaret`/`cm-ySelectionCaretDot`/`cm-ySelectionInfo` classes that were never defined anywhere in the codebase — not even in local dev — so remote cursors would have been invisible/degenerate (a `span` with block-level children and no CSS collapses to a giant zero-visual-distinction box) regardless of the packaging bug above.
   - **Fix**: added real layout CSS for all three `cm-y*` classes to `packages/plugin/styles.css`, and added `packages/plugin/styles.css` to the `files:` list in `plugin-release.yml`.
   - **Verified live** (not just by reading code): built the exact DOM structure `folder-presence.ts`/`remote-selections.ts` create in a real running Obsidian (Electron/Chromium) window via CDP `Runtime.evaluate`, and measured `getBoundingClientRect()` with no CSS loaded (matching the current broken production release) vs. with the fixed `styles.css` injected. Before: presence dot `0×0`; cursor caret/dot/name-tag collapsed into an `800×19.5` block with no distinguishable shape. After: presence dot `10×5`, caret `3.2×20` (a proper thin cursor line), caret dot `8×8`, name tag `26.5×15.6` — all match the new CSS rules exactly.
   - **Shipped** as `0.1.5`. That isolated-DOM sizing check only proved the caret/dot/name-tag had *non-zero layout* — it built the widget's DOM in isolation with hand-picked colors and never exercised two real clients exchanging awareness over the network, so it couldn't have caught (and didn't catch) the two further bugs found in the very next round of testing — see the section below.
3. **Incidental fix**: added a `user set-password <username> --password <pw>` CLI command (`db.ts`'s `setPasswordHash` + `cli.ts`) — there was previously no way to reset a dev/test account's password (only `user add`/`user delete` existed), which blocked re-authenticating `alice`/`bob` in the local dev vaults after their session tokens expired. Also bumps `token_version` like `user revoke` does, since a password change should invalidate old sessions.

## CI + release-asset guard (✅ done) — why the above bugs reached production undetected

Both Phase 7 continued bugs above reached a real production deployment despite a solid `verify:*` test suite already existing, for two distinct structural reasons — this section closes both gaps, not just the two bugs themselves:

1. **Nothing ran the test suite automatically.** Every `verify:*` script (and `sync-core`'s `verify`) only ever ran when a human happened to type the command by hand. There was no CI workflow triggered on push/PR at all — only `docker-publish.yml` and `plugin-release.yml`, both tag/branch-triggered release pipelines, not test gates.
   - **Fix**: `.github/workflows/ci.yml` (new) runs on every push and pull request to `main` — builds all three packages, then runs `pnpm run test` (see below). Any future regression covered by these tests (including a reintroduction of the Y.Map data-loss bug) now fails a PR automatically instead of silently shipping.
   - **`pnpm test`** (repo-root `package.json`, new script) runs the entire verify suite — `sync-core`'s `verify`, and the server's `verify:sync`/`verify:persistence`/`verify:auth`/`verify:attachments`/`verify:admin`/`verify:rooms`/`verify:offline-merge` — in one command. This is exactly what CI calls, and is the one thing to run locally before considering any change done.
2. **A GitHub Release's shipped-file list was a second, hand-maintained copy of "what should ship," independent of what actually existed on disk.** No app-level test could have caught the missing-`styles.css` bug — it's a packaging/release-artifact problem, not a runtime behavior problem.
   - **Fix**: `plugin-release.yml`'s "Verify required plugin assets exist" step defines `REQUIRED_FILES` exactly once, asserts every file in it exists (failing the release loudly if not), and passes that *same* variable to `gh release create` — the existence check and the actual upload can no longer drift apart, because there's only one list.

**Before shipping a new plugin release**, in order: (1) `pnpm test` passes locally, (2) `packages/plugin/manifest.json`'s `version` is bumped, (3) if a new file needs to ship (beyond `main.js`/`manifest.json`/`styles.css`), update `REQUIRED_FILES` in `plugin-release.yml` accordingly, (4) tag with that exact bare version string and push the tag.

## Phase 7 continued, round 2 — remote selection highlight had invisible color; cursor name-tag couldn't be hovered (✅ fixed, verified with two real synced CM6 editors)

Requested as a follow-up once presence dots/cursors were confirmed shipping correctly: make text *selections* (not just the cursor position) visible on other users' screens, and make the remote cursor's name tag appear on hover. Both were already implemented in `remote-selections.ts`/`styles.css`, but each had one concrete bug that made them non-functional — found only once tested with two *real* CM6 editors exchanging real awareness state over a real Hocuspocus connection (the previous round's isolated single-DOM CSS check couldn't have caught either):

1. **The selection highlight's color was always invalid CSS, so it rendered fully transparent.** `colorLight` (the translucent highlight color) fell back to `` color + "33" `` — a hex-alpha-suffix trick that only produces valid CSS when `color` is a hex string. `colorForUsername()` (`room-manager.ts`) returns `hsl(hue, 70%, 45%)`, so the fallback produced `hsl(275, 70%, 45%)33` — a string the browser silently rejects, confirmed directly via `getComputedStyle(...).backgroundColor` reporting `rgba(0, 0, 0, 0)` even though the decoration, its `class="cm-ySelection"`, and its (invalid) inline style were all present and correct in the DOM.
   - **Fix**: replaced the fallback with `` `color-mix(in srgb, ${color} 25%, transparent)` `` (`remote-selections.ts`) — works for any valid CSS color syntax (hex, hsl, rgb, named), so it can't silently break again just because `colorForUsername`'s output format changes.
2. **The cursor's name tag could never appear on hover, in any browser, ever.** `.cm-ySelectionCaret` had `pointer-events: none` — which means the element **cannot receive pointer/hover events at all**, so the `.cm-ySelectionCaret:hover .cm-ySelectionInfo { opacity: 1 }` rule was permanently dead code, confirmed by checking `caret.matches(':hover')` after a real CDP-simulated mouse move (`Input.dispatchMouseEvent`) landed directly on the caret's bounding box — `false`, and the name tag's opacity stayed `0`.
   - **Fix**: changed `pointer-events` to `auto` and added a few px of transparent horizontal padding (`styles.css`) so a real 3px-wide caret bar is a practical target to hover, not just a technically-hoverable one.
- **Verified with two real synced CM6 editors** (not Obsidian — a small standalone harness, since `remote-selections.ts` has zero Obsidian dependencies): esbuild-bundled `EditorState`/`EditorView`/`yCollab`/`HocuspocusProvider`/`hardenedRemoteSelections` into one page, loaded via CDP into a real Chromium (an isolated Electron/Obsidian instance used purely as a browser host — no vault involved), two `HocuspocusProvider`s connected to the local dev server's real `shared` room with real `alice`/`bob` JWTs. Confirmed: alice selecting text produces a `.cm-ySelection` in bob's DOM with a real, non-transparent computed `background-color`; a real CDP mouse move onto bob's rendered caret makes `:hover` match and the name-tag opacity animate `0 → 1`.
- **Gotcha hit while building the test harness itself**: the harness's own two-editor setup first hit the *exact* Y.Map collision bug from the round-1 fix above — both editors called `getOrCreateFileText` immediately after constructing their providers, before either had synced, so each created its own colliding `Y.Text` for the same path and bob's editor got permanently stuck on an orphaned object. Fixed by awaiting `provider.isSynced`/the `"synced"` event before touching the doc at all — direct, live re-confirmation that hydrate/sync-before-touch is the correct general rule, not something specific to `y-indexeddb`.
- **Follow-up found in real use, right after shipping the two fixes above**: the caret rendered as a "ridiculously thick" solid block instead of a thin line. Two compounding causes, both introduced by the round-1 CSS fix: (1) the border was `2px` on each side — double `y-codemirror.next`'s own reference implementation's `1px`; (2) `YRemoteCaretWidget.toDOM()` set `background-color` on the *whole wrapping span* (so the dot/name-tag could `inherit` it), but that background paints the **entire box including the transparent hover-hit-area padding** added in the same fix — so the "invisible wider hit target" was actually a solid `~10px` colored block. Fixed by having the dot and name-tag set their own `background-color` directly (no longer relying on `inherit`), leaving the wrap itself with no background at all — only its 1px borders are visibly colored, and the padding around it is now genuinely transparent. Verified via the same CDP DOM-injection technique: wrap's computed `background-color` is `rgba(0,0,0,0)`, dot/info still show the correct color, total visible width dropped from a solid block down to a normal thin line.
- **Second follow-up, from an actual screenshot**: the caret's dot wasn't centered over the line, and a *second* vertical line was visible to the right. First diagnosis (incomplete — see third follow-up below): `.cm-ySelectionCaret` still had both `border-left` *and* `border-right` in **our own** `styles.css` — dropped `border-right` there and recomputed `.cm-ySelectionCaretDot`'s `left` for a single border. Verified numerically via CDP (`getBoundingClientRect()` on both the border and the dot, `dotCenterX - borderLineCenterX` landed at `-0.2px`) — but this fix alone turned out not to be sufficient in real use (see next entry).
- **Third follow-up — the actual root cause of the "second line" surviving the previous fix**: user reported the double-cursor was still there after reloading. The real bug: `y-codemirror.next`'s `yCollab()` pushes **two separate extensions** for remote cursors — the `yRemoteSelections` `ViewPlugin` (correctly filtered out in `main.ts` since Phase 1/round-1) **and** a distinct `yRemoteSelectionsTheme` (a CM6 `EditorView.baseTheme(...)`) that was never filtered. That theme defines its own `.cm-ySelectionCaret` CSS — including its own `border-right: 1px solid black` — completely independently of our `styles.css`. Since it was still active, editing *our* CSS's border-right had no effect: the browser was still applying the theme's own still-injected rule. Fixed by also excluding `yRemoteSelectionsTheme` from the extensions in `main.ts`'s `bindIfTarget()`. Verified precisely (not just by re-reading the diff): CM6's `baseTheme` uses `style-mod`'s `StyleModule`, which injects a real `<style>` sheet into the document **the first time an editor using it mounts, and never removes it** — so a naive single-page test that tries "old filter" then "new filter" back-to-back gives a false pass, since the theme's stylesheet from the *first* editor lingers and contaminates the check for the second. Correct test: reload to a genuinely fresh page for each variant, then inspect `document.styleSheets` for the theme's specific fingerprint (`.cm-ySelectionCaretDot` with `.4em`, a value only the *upstream* theme uses, never our own `8px`). Old filter → fingerprint present; new filter (excluding the theme too) → fingerprint absent, on a clean page each time.
- **Fourth follow-up — "spacing gets wonky while selecting on the other client"**: extensive live testing (line-height measurements, mark-decoration color checks, character-by-character simulated drags) found no dramatic layout bug — the actual, measurable effect was the horizontal padding added to `.cm-ySelectionCaret` in round 1 (for a bigger hover target) visibly shoving adjacent letters apart by a few px whenever the widget lands mid-word, which happens constantly while a remote user is actively dragging a selection. That padding was never part of upstream's design and had already caused two earlier bugs in this same area (the thick-block and double-border follow-ups above) — removed it entirely rather than tune it further, matching upstream's un-padded caret exactly. Also restored `.cm-yLineSelection`'s `padding: 0; margin: 0 2px 0 4px;` (verified via CDP: was completely unstyled after `yRemoteSelectionsTheme` got excluded in the third follow-up, since nothing in our own `styles.css` ever defined it — a real, if minor, regression from that fix that hadn't been caught yet). Separately confirmed structurally that a real native Obsidian vault cannot be scripted open via CDP for further live verification in this environment: "Open folder as vault" invokes Electron's `dialog.showOpenDialog` from the main process, a genuine OS-level dialog outside the page/DOM model entirely — CDP's `Page.setInterceptFileChooserDialog` only intercepts HTML `<input type="file">` pickers, a fundamentally different mechanism, so this isn't a workaround-able tooling gap.

## Key gotchas discovered (do not re-debug these from scratch)

1. **`registerEditorExtension` doesn't retroactively apply to already-open editors.** Obsidian restores the last-open file *after* plugin `onload()` returns, not before. Fix: `workspace.onLayoutReady()` — but also...
2. **`onLayoutReady()` can be slow or (in edge cases observed directly) never fire.** Never let plugin functionality depend on it without a timeout fallback.
3. **Never dispatch to a CM6 `EditorView` synchronously from inside a `ViewPlugin` constructor.** It runs mid-update; CM6 throws "Calls to EditorView.update are not allowed while an update is in progress" and silently drops the change. Defer with `queueMicrotask`.
4. **`Y.Map.observe()` is shallow — use `observeDeep()`.** A new file is two separate Yjs changes (key added with empty `Y.Text`, then content inserted into it). Shallow `observe()` only fires for the first; the content update needs `observeDeep()`, and the handler must use `event.path` to find nested changes (not just `event.keysChanged`).
5. **Creating a new `Y.Text` entry and inserting its content must be ONE atomic transaction**, or your own change-observer can fire twice (empty, then real) and the two async disk-writes can race and land out of order — this actually happened (a renamed file ended up empty on disk). Always use `setFileContent(doc, path, content)` from `sync-core`, never the raw `getOrCreateFileText` + `reconcileYTextWithContent` two-step.
6. **`onload()` must never block on network/layout conditions.** Obsidian appears to await each plugin's `onload()` during its own startup — an unresolved promise in there (e.g. waiting on an unreachable server with no timeout) hangs *all of Obsidian*, not just the plugin. This actually happened during development. Pattern: `onload()` does synchronous setup only, then kicks off `void this.initializeAfterLoad()` (not awaited) for anything that touches the network or `onLayoutReady`.
7. **Two independently-created Yjs values assigned to the same `Y.Map` key do not merge — one silently discards the other entirely.** This is not specific to `Y.Text`/files; it's how `Y.Map` conflict resolution works for any nested shared type. Never call `filesMap.set(path, new Y.Text())` (or the equivalent `getOrCreateFileText` pattern) on a `Y.Doc` that hasn't yet confirmed whether that key already exists elsewhere (i.e. before a sync/hydration has happened) — see the Phase 7 continued section above for the full mechanism and fix (local `IndexedDB` persistence via `y-indexeddb`, hydrated before anything else touches the doc).
8. **A GitHub Release's attached files are exactly what BRAT ships — nothing else, regardless of what's in the repo or the build output.** `plugin-release.yml`'s `files:` list is the actual contract; a CSS file (or anything else) sitting correctly in `packages/plugin/` after `esbuild` means nothing to a BRAT install if it isn't in that list. Local dev testing via `sync-test-vaults.mjs` copies everything unconditionally, so this class of bug is invisible locally — always check the release workflow's file list directly when a feature "works locally but not for BRAT users."
9. **A linked room's "local folder" setting (`vaultFolder`) must be a vault-relative path, never an OS filesystem path.** Obsidian's own `TFile.path` values are always relative to the vault root (e.g. `Test Folder/test.md`); `isUnderFolder`/`FileSyncEngine` compare directly against that, with no normalization. Pasting a real absolute path (e.g. copied straight from File Explorer, `A:\Coding\...\Test Folder`) into the settings-tab text field silently breaks sync — no error surfaces anywhere, the plugin just believes there are zero local files under the target and never syncs anything. This was the actual root cause of a real "I linked a room but nothing is syncing" report. Fix is the bare relative folder name only (e.g. `Test Folder`). Editing `data.json` directly also works, but a currently-running Obsidian instance holds settings in memory and won't notice the on-disk change until the vault/plugin is reloaded.

### Testing methodology gotchas (only relevant if automating Obsidian via Chrome DevTools Protocol like this session did)

Obsidian is Electron, so it can be launched with `--remote-debugging-port=<port> --user-data-dir=<isolated dir>` and driven via CDP (`ws://localhost:<port>/json` for targets, then `Runtime.evaluate` over the target's websocket) — this is how every Phase 1/2 test in this project was actually verified end-to-end without the user needing to click anything. Gotchas specific to this approach, not to Obsidian itself:
- A fresh isolated profile needs `app.plugins.setEnable(true)` called once via CDP (community plugins are off by default; `community-plugins.json` alone isn't enough).
- A fresh/automated launch can leave `app.vault.getFiles()` returning `[]` and vault events never firing, even though the adapter can see real files on disk (`app.vault.adapter.list('')` works fine) — fix is calling `app.vault.load()` once via CDP. This is an artifact of the automated launch path, not a real bug.
- Prefer a full process relaunch over `location.reload()` when testing vault-event-dependent behavior — reload can leave `workspace.layoutReady` stuck `false` and vault events dead, seemingly tied to corrupted profile/workspace state from repeated in-page reloads.
- Use `require('electron').remote.app.quit()` via CDP to cleanly quit an isolated instance (confirm via `Get-CimInstance Win32_Process` filtered to the custom `--user-data-dir` before assuming it's gone — Electron spawns several child processes per instance).
- **Opening a vault by path via CDP is unreliable** — a fresh/isolated profile shows Obsidian's own "starter" screen (`app://obsidian.md/starter.html`), and its "Open folder as vault" button triggers a *native* OS folder-picker dialog that CDP cannot drive (it's outside the page DOM entirely). Hand-writing that profile's own `obsidian.json` vault registry ahead of time (mirroring the real one's `{"vaults":{"<id>":{"path":...,"open":true}}}` shape) does *not* reliably force an auto-open either. For CSS/DOM-only verification you don't actually need a real vault at all: connect CDP to the starter screen itself (it's still a full Chromium page) and build the exact DOM structure your widget/badge code creates via `Runtime.evaluate`, inject the real `styles.css` content as a `<style>` tag, and read `getBoundingClientRect()`/`getComputedStyle()` — this is how the presence-dot/remote-cursor CSS fix was verified (0×0 with no CSS → correct pixel dimensions with the fixed CSS), entirely without a working vault-open flow.
- **For anything that needs two real, *synced* clients (not just static DOM/CSS), skip Obsidian entirely and load the actual CM6/Yjs modules in a plain page.** `remote-selections.ts` (and anything else with no `obsidian` import) can be esbuild-bundled standalone (`bundle: true, format: "iife"`, no `external` list, unlike the real plugin build which externalizes `@codemirror/*` for Obsidian to provide) into a tiny test page with two mount divs, each running its own `EditorView` + `yCollab` + `HocuspocusProvider` connected to the real local dev server. Load that page into the same CDP-reachable Chromium (`Page.navigate` to a `file://` or local static-server URL works on the *same* target used for the starter-screen technique above — no vault-open needed at all). This is how selection-highlight sync and the cursor-hover name tag were verified against two real, independently-synced editors instead of one hand-built DOM.
- **A real CSS `:hover` state can be verified with CDP** without any human present: get the target element's `getBoundingClientRect()`, send `Input.dispatchMouseEvent` with `type: "mouseMoved"` at its center, then check `element.matches(':hover')` and `getComputedStyle(...)` — this is a genuine browser hover, not a class toggle, so it also catches `pointer-events: none` mistakes (an element with that set will never satisfy `:hover` no matter where the mouse moves, exactly how the cursor-name-tag bug was found).
- **When testing multiple independent Yjs clients against the same fresh room path, always await `provider.isSynced`/the `"synced"` event before calling `getOrCreateFileText`/`setFileContent` on any of them** — calling it immediately after construction (before the network round-trip completes) reproduces the Y.Map same-key-collision data-loss bug (see the Phase 7 continued section above) even in a two-line test harness, not just in the real plugin across an app restart.
- **CM6's `EditorView.baseTheme()` (via `style-mod`'s `StyleModule`) injects a real `<style>` sheet the first time any editor using it mounts, and never removes it — not even after that editor is `.destroy()`'d.** A test that tries variant A (theme included) then variant B (theme excluded) on the *same* page will find variant A's injected CSS still present when checking variant B, giving a false "still broken" result that has nothing to do with variant B's actual correctness. Always reload to a genuinely fresh page (`Page.navigate`) between variants when checking for the presence/absence of CM6-theme-injected CSS via `document.styleSheets`.
